## 二、`image_generate` 工具的模型路由指导

工具 schema 的 description 明确定义了三个模型的适用场景：

| 模型 | provider | 适用场景 |
|------|----------|---------|
| **Gemini** | `gemini` | **consistency-preserving reference workflows**、realistic rendering、image-to-image iteration、**packaging/key-frame generation**。当已有可靠的视觉参考且目标是保持稳定地扩展/适配/精修时优先选用。 |
| **Seedream** | `seedream` | high-aesthetic **hero visuals**、fine-detail rendering、polished **brand/product imagery**。 |
| **GPT-image-2** | `fal` | fine-grained natural language edits、in-painting、text-heavy illustrations requiring fine typography。 |

## 三、逐 case 分析

### Case 11834755817 — 诊所环境参考图

**用户原话**：
> 故事发生在一间「老式社区小诊所候诊室」……请生成一张**环境参考图**：竖屏 9:16。画面要能看出空间——塑料联排椅、贴墙的通知栏……不要具体剧情人物占满画面……环境信息要足够具体

**为什么应该用 Gemini**：
- 用户明确说这是一张「环境参考图」→ 对应工具描述中 Gemini 的 **「key-frame generation」**
- 目的是为后续视频制作提供场景锚点 → 对应 **「consistency-preserving reference workflows」**
- 强调写实空间感（塑料椅、地砖、油漆）→ 对应 **「realistic rendering」**
- 不需要高美感/品牌感 → **不匹配 Seedream** 的定位

**Agent 行为**：未读取 skill 文件，直接 `write_todos` → `image_generate(seedream)` → `add_attachments`

### Case 46602679325 — 风格参考帧/锚点

**用户原话**：
> 生成一张「纯风格参考帧」……这张图**只作为后续所有镜头的色彩、颗粒与光线质感的锚点**，构图可以简单，但色与光要稳定可延续

**为什么应该用 Gemini**：
- 用户**明确说**这是「锚点」，目标是**稳定可延续** → 完美匹配 Gemini 的 **「consistency-preserving」** 和 **「keeping the result stable」**
- 这是一张参考帧（key-frame）→ 对应 **「key-frame generation」**
- 目标不是独立的高美感产出，而是**可供后续一致性复用的基准** → 不匹配 Seedream 的 hero visuals 定位

**Agent 行为**：未读取 skill 文件，直接 `write_todos` → `image_generate(seedream)` → `add_attachments`

### Case 73418744404 — 咖啡品牌 5 风格探索

**用户原话**：
> 做一个咖啡品牌的视觉**风格探索**，请帮我生成**5 个不同风格方向**的咖啡品牌视觉概念图，我看看**哪个方向好**

- 5 张并发生成 → 效率优先 → Gemini 更合适

**Agent 行为**：读取了 `generation-skill/SKILL.md`（包含 Rule 5 和 Rule 6），但仍然选择了 Seedream ×5

## 四、共性根因

### 4.1 Seedream 被当作默认模型

三个 case 中，Agent 均**无条件选择 Seedream**，没有做模型路由判断。表现为：
- Case 11834755817、46602679325：**跳过 skill 文件读取**，直接调用 Seedream
- Case 73418744404：**读了 skill 文件**（含 Rule 5 & 6），但仍然选择 Seedream — 说明规则被忽略

### 4.2 「参考帧 / key-frame / 锚点」场景的路由信号未被识别

工具描述中 Gemini 的两个关键标签：
1. **「key-frame generation」** — 当用户说「参考图」「参考帧」「锚点」时应匹配
2. **「consistency-preserving reference workflows」** — 当用户说「稳定可延续」「后续所有镜头的锚点」时应匹配

Agent 没有识别这些信号。

### 4.3 generation-skill Rule 5 & 6 形同虚设

Rule 5 要求「broad exploration → 高效模型优先」，Rule 6 要求「Seedream 仅限动画/CG/插画」。但即使 Agent 读了 skill 文件（case 73418744404），也没有执行这两条规则。

## 五、修复建议

### 5.1 短期 — 强化路由信号识别

在 `image_generate` 工具描述或 generation-skill 中增加**显式的路由决策树**：

```
用户意图是什么？
├── 参考帧 / 关键帧 / 场景锚点 / 风格锚点 / storyboard
│   └── → Gemini（consistency-preserving + key-frame generation）
├── 风格探索 / 概念草案 / 多方向对比
│   └── → Gemini（efficient broad exploration，Rule 5）
├── 最终交付的品牌/产品 hero visual
│   └── → Seedream
├── 动画 / CG / 插画 / 二次元
│   └── → Seedream（style-specialized，Rule 6）
├── 文字密集 / 排版 / 信息图
│   └── → GPT-image-2
└── 精细编辑 / in-painting / 区域修改
    └── → GPT-image-2
```

### 5.2 中期 — 新增 `references/image-models-scheduling.md`

类似视频生成的 `references/video-models-scheduling.md`，为图片生成提供结构化的：
- 模型能力矩阵（三模型对照表）
- 基于用户意图关键词的路由映射
- 场景示例

### 5.3 长期 — PatchToolCallsMiddleware 增加图片路由检查

在 middleware 层面检测常见的路由错误：
- prompt/用户请求中包含「参考」「锚点」「key frame」「storyboard」→ 建议或强制 Gemini
- prompt 中包含大量文字渲染需求 → 建议 GPT-image-2
- 非动画/CG/插画场景使用 Seedream → 警告
