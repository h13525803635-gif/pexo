核心结论 TL;DR

kling 路由不在预期范围内。用户「主播 + 口播 + 中文配音」的关键词组合，按已沉淀的标准产线（case 04453569710、96933223677）应当路由到 seedance reference2video + audio_list (TTS) + sound:"on"。本 case 的 Agent 跳过了路由决策层（generation-skill / script-skill 一次都没读），凭先验直接选了 kling-v3-omni reference2video，并且 没传 audio_list、保留 sound:"on"，导致模型自由 co-gen 出英文人声、嘴型零驱动。后续装配阶段又违反 assembly-skill 硬规则手算字幕时间，并把英文 co-gen 人声压到 0.2 而不是归零，最终交付时对所有这些差异零字披露。

关键失败链 Causal Chain

用户口播需求 ↓ Agent 跳过 generation-skill / script-skill 路由层 ↓ 凭先验选 kling-v3-omni reference2video（默认产线，不是口播产线） ↓ 不传 audio_list（嘴型零驱动） + sound:"on"（让模型自己 co-gen 英文人声） ↓ TTS 独立做 → 三套时间轴互不相干（嘴 vs 中文配音 vs 残留英文） ↓ 装配阶段没归零视频音轨（保留 0.2 英文人声） ↓ 字幕手算时间（无视 ElevenLabs alignment json，违反 assembly-skill 硬规则） ↓ 时长不够用同一段重复 5s 凑齐 ↓ 交付时一句风险都不提
