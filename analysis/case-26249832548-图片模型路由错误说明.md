# 项目 26249832548：image_generate 模型路由错误 — 应用 GPT-image 却用了 Seedream

**Langfuse trace**：`analysis/langfuse-data/cases/26249832548/trace-1-5423ead2.json`  
**分析日期**：2026-05-07  

---

## 一、用户侧现象

用户请求生成一张**数据可视化 MG 动画关键帧**，要求画面显示中文标题「2025年全球AI市场规模」、柱状图、以及数字标注「$180B」。这是一个**文字密集 + 排版密集**的信息图场景。

---

## 二、实际调用

```json
{
  "name": "mg_keyframe_01",
  "provider": "seedream",
  "model": "seedream-4-5-251128",
  "mode": "text2image",
  "prompt": "...Centered bold white title text '2025年全球AI市场规模'... the tallest bar prominently labeled '$180B'..."
}
```

Agent 选择了 **Seedream** 作为图片生成模型。

---

## 三、根因：违反了 generation-skill 规则 + 工具描述

### 3.1 generation-skill 明确规定的路由原则

generation-skill `Direct Image / Content Generation Mode` 中有两条直接相关的规则：

> **Rule 5** — Model efficiency matters: Default toward efficient broad exploration first; **reserve slower high-control paths for text-heavy, layout-heavy, or precision-edit cases.**

> **Rule 6** — **Seedream is style-specialized**: Prefer Seedream **only** when the request explicitly points to anime, animation, CG, illustration, or comparable stylized visual worlds — **not as the generic beauty default.**

本 case 的用户需求：
- **文字密集**：中文标题 + 英文数字标注
- **排版密集**：柱状图、网格线、坐标轴标签
- **非动画/CG/插画风格**：科技感信息图，属于 UI/数据可视化

按 Rule 5 + Rule 6，Seedream **不应被选中**。

### 3.2 `image_generate` 工具描述中的模型指导

工具 schema 的 description 字段明确写了三个模型的适用场景：

| 模型 | provider | 适用场景 |
|------|----------|---------|
| **Gemini** | `gemini` | 一致性保持的参考工作流、写实渲染、image-to-image 迭代、包装/关键帧生成 |
| **Seedream** | `seedream` | 高美感视觉、精细渲染、品牌/产品图、更精致的视觉效果 |
| **GPT-image-2** | `fal` | **fine-grained natural language edits、text-heavy illustrations requiring fine typography** |

用户的需求是**含大量中英文文字的信息图/数据可视化**，明确匹配 GPT-image-2 的描述：「**text-heavy illustrations requiring fine typography**」。

### 3.3 Agent 为什么选错了

1. **未读取 generation-skill**：Agent 收到用户请求后，**跳过了 skill 文件读取**，直接调用 `write_todos` → `image_generate`。工具调用序列：
   ```
   [1] write_todos
   [3] image_generate  ← 未先读取 generation-skill/SKILL.md
   [5] write_todos
   [7] add_attachments
   ```
   
2. **忽略工具描述中的模型指导**：即使不读 skill 文件，`image_generate` 工具的 description 中已经写明了三个模型的路由逻辑，Agent 仍然选择了不匹配的 Seedream。

---

## 四、正确的调用应该是

```json
{
  "name": "mg_keyframe_01",
  "provider": "fal",
  "model": "openai/gpt-image-2",
  "mode": "text2image",
  "provider_param": {
    "fal_gptimage2_text2image": {
      "prompt": "...2025年全球AI市场规模...$180B...",
      "aspect_ratio": "16:9"
    }
  }
}
```

理由：
- GPT-image-2 专为「text-heavy illustrations requiring fine typography」设计
- 该需求的核心是**文字清晰可读 + 信息图排版**，不是美学风格化

---

## 五、修复建议

### 5.1 Agent 行为层面

1. **强制 skill 读取**：对于 `image_generate` 调用，Agent 应在首次生成前读取 `generation-skill/SKILL.md`，尤其是 Rule 5 和 Rule 6。当前 Agent 跳过了 skill 读取直接生成。

2. **工具描述遵从**：即使跳过 skill 文件，Agent 也应遵从工具 description 中已有的模型指导。当前存在 "看到了工具说明但未执行" 的问题。

### 5.2 Skill / 系统层面

3. **image-models-scheduling 参考文档缺失**：视频生成有 `references/video-models-scheduling.md` 做详细的模型路由决策树，但**图片生成没有对应的 `references/image-models-scheduling.md`**。图片模型路由规则分散在：
   - 工具 description（简要）
   - generation-skill Rule 5/6（原则性）
   
   没有像视频那样的结构化决策树。建议新增 `references/image-models-scheduling.md`，包含：
   - 三模型适用场景对照表
   - 基于需求特征的决策树（文字密集→GPT-image / 风格化→Seedream / 参考一致性→Gemini）

4. **PatchToolCallsMiddleware 可强化**：在 middleware 层面可增加检查逻辑 — 当 prompt 中包含可识别的文字渲染需求（中文标题、数字标注等）时，提示或自动修正 provider 为 `fal`/GPT-image-2。
