## 一、用户侧现象

- 字幕**层叠/糊在一起**（同屏两行）、或**切换不干脆**。  
- 希望字幕**只**在念旁白时出现；在**无旁白**的片段字仍不消失。  

---

## 二、先分清：你说的「重叠」是哪一种

| 现象 | 含义 | 与下文的对应 |
|------|------|--------------|
| **A. 两行书同时出现在画面上** | 时间轴上**两段子幕 clip 的区间有交集** | 见第三节 第 1 点 |
| **B. 只有一行字，但上句没结束下句声音已开始/字「粘」在一起** | 多数字幕不是物理叠两层，而是**上一条 `out_ms` 过晚**或**下一条 `start_ms` 偏早** | 见第三节 第 2 点 |
| **C. 画面里已有大段 UI/图表字，再叠底栏字幕** | 视觉拥挤，**像**叠字；本质是两路**文字源** | 见第三节 第 3 点 |

本 trace 中 Agent 使用 **`kind: "text"`** 手写每行 `start_ms` / `out_ms`；**重叠问题主要是 A 或 B 的组合**，与「两路 TTS/片内声叠音」不是同一类根因。

## 三、字幕为什么会重叠

### 1. 真重叠：两条 `text` clip 的时间窗「相交」

`execute_edit_video` 的 **文本轨**在渲染时，会在**全局时间线**上按每条 clip 的 `start_ms` / `out_ms` 开画、停画。  
**若**两条 clip 在半开区间 `[start_ms, out_ms)` 上存在**非空交集**（即存在某一时刻，两行都处于「应显示」的区间里），则同一时刻**两行都被绘制** → 用户看到的就是**真·字叠字**（常见为上下两行同屏、或一段时间里两行重叠）。

- **本项目中为什么会发生？**  
  当前成片**没有**使用 TTS 返回的 **`subtitle_file` 里每个 cue 的精确起止**；时间轴是**按文意分句后人工填毫秒**的。只要某两行边界算错、或「匀时长」时把相邻行窗口挤到交叉，就会触发 **A 类重叠**。  
- **和「对不齐」的关系**：对不齐是「字与声」的错位；**重叠是「行与行」的区间在时间上打架**，二者可同时存在。  

**自检规则（在 skill 里应写清）**：对相邻两行、若本不应同屏，应满足 `out_ms[上一行] <= start_ms[下一行]`（或等价地保证两段区间无交集）。**禁止**在不该双行同显时，出现 `out_ms[上一行] > start_ms[下一行]`。

### 2. 假重叠/拖影：只有一行在画，但**停得太晚**或**起得太早**

即便任意两行区间**不交叉**，若：

- 上一行 **`out_ms` 明显晚于**该行旁白在音频里**实际结束**的时刻，或  
- 下一行 **`start_ms` 早于**下一句 TTS 真正开声的时间，

听感上会像「上句还赖在屏上/上下句顶在一起」，部分用户也会口头说成「叠在一起」。**根因仍是手填时间与真实发声不一致**——本 trace 中旁白有 API 的 **`duration`** 与 **`subtitle_file`** 可用，但成片未用其驱动 `text` 轨。

**与 A 的区分**：此时同一瞬间通常**不是**两段子幕 clip 同时 `enable`，而是**单条**停留过久或切换偏晚。  

### 3. 与「画内字」的混淆

若 `video_generate` 画面前景里本来就有**清晰的英文 UI、图表、标题**，再叠一层底部烧录字幕，视觉上会**很挤**；这**不是**文轨上两条 clip 时间相交，但报告问题时常被笼统说成「字重叠」——需与 1 区分。  

### 4. 首次 `subtitle` 轨 + MP3 为何不是本问题的正解

Agent 曾用 **`kind: "subtitle"`** + 各段 **MP3 路径** + 仅 **`start_ms`**，**缺少** **`out_ms`**。渲染器报：`out_ms not set and no probe function available`，即**在 DSL 转渲染计划时无法确定每段烧录的结束点**，**无法**可靠地「一行对应一声」。失败后改为手搓 `text` 轨，**并未**自动消除重叠，只是把错误从「系统拒渲染」变成「人填时间窗的精度问题」。

---

## 四、其它问题：不消失、与旁白不同期（简要）

- **不消失、无旁白时仍有字**：多为主因 **第三节第 2 点**——`out_ms` 拖过真实发声结束。  
- **与旁白不同期**：`subtitle_file` 与 **`duration`** 未作为真源，手写行级时间与 TTS 实际节奏不一致。  
- 本 trace 中另有 **BGM 轨编排** 导致首版成片时长异常拉长等，**属音频轨问题，不是字幕行重叠的同一根因**。

## 五、对 assembly-skill 的修改建议（可落地）

以下为建议写入 **`assembly-skill/SKILL.md`** 或独立 **`references/assembly-subtitles.md`**（由 SKILL 引用）的条文化内容；重点补上 **「行与行不得时间相交 + 以 subtitle_file 为真源」**，避免第三节中所述重叠问题反复出现。

### 1. 时间轴真源（硬规则）

- 凡经 **`audio_produce`** 的旁白，**必须优先**用返回的 **`subtitle_file`（JSON）** 生成烧录或字幕轨。  
- **禁止**以「将全文按句均分毫秒」作为唯一手段；若 `subtitle_file` 不可用，须以 **ffprobe 实测**每段 MP3 的 `format.duration` 为边界，并显式约束每句的 `out_ms`。

### 2. 多段 VO 与字幕对齐

- 每条 cue 的**时间线** = 该段 VO 在 `execute_edit_video` 中的 **`start_ms`** + cue 在 `subtitle_file` 内的**相对时间**（若产品约定 JSON 为绝对时间，应在 skill 中**写死一种**换算方式）。  
- 与既有「**相邻 VO 的 start 由 ffprobe 逐条累加**」规则一致，避免与音轨错位。

### 3. DSL 约束

- **`kind: "subtitle"`** 须满足渲染器对 **`out_ms`（或等价）** 的要求，不得仅 `file: *.mp3` + `start_ms` 且无结束时间。  
- 烧录**推荐**来自 **解析 `subtitle_file` 后的 cue 表**，而不是把 MP3 当作未定义长度的默认字幕源。

### 4. 明确禁止

- 禁止：无 `subtitle_file`/duration 支撑的手拍「均匀切分」**`text` 轨**作为默认方案。  
- 禁止：相邻两行 **时间窗未校验** 即提交（应满足不叠读时 `out_ms[i] <= start_ms[i+1]` 等关系）。

### 5. 调用前自检

- 每段 VO 的 **duration** 已来自 API 或 ffprobe。  
- 每行字幕的区间落在此段 VO 的有声区间内；段间**静音**无 clip。  
- 若存在多行 **同一时段不应同屏** 的文案，**任意两行之区间不得相交**。

### 6. 与画面内字

- 需求为「清晰可读英语字幕」时，以**后期烧录/轨**为主，并控制画内大段正文字/UI 的密度，减少与第三节第 3 点所述现象混淆。  

### 7. 交叉引用

- 在 `references/video-models-scheduling.md` 的 Post-assembly 小节，加一句指向「字幕/烧录真源与行级时间窗校验」。
