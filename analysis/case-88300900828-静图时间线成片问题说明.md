# Case 88300900828：只生静图、用时间线成片的原因与解决办法

**Langfuse trace**：`pexo:88300900828`（`trace-1-ab7c6fa8.json`）  
**会话时间**：2026-05-14（production）

---

## 现象

- 用户期望偏「短视频 / 多场景成片」，但 Agent **大量调用 `image_generate`**，**未调用 `video_generate`**。
- 成片路径为：**静图 + 黑底时间线 + overlay + 字幕/配音/BGM**，通过 `video-editor__execute_edit_video` 组装。
- 从工具调用看：**不是**视频模型报错后的回退，而是模型**主动选择**的制片策略（trace 中有明确文案：用 images 做 slide-style clips）。

---

## 原因总结

### 1. 用户稿形态易被解读为「配图」而非「运动镜头」

脚本结构为「Scene + **Picture to show** + 口播 + Caption」。  
「Picture」在语义上更接近**静态画面 / 信息图**，Agent 容易判断为：**每段一张图 + 口播**即可满足 brief，无需每段 `video_generate`。

### 2. 内容类型强化「静帧够用」的判断

分子示意、反应式、方程式板书等 **教育 / 信息图** 场景，在常识上更像 **PPT / slide explainer**，与「静图 + 字幕 + VO」模板高度一致。

### 3. 成本与路径复杂度

并行 `audio_produce` + `image_generate` + `music_generate`，再一次性 `execute_edit_video`：

- 避免多段 `video_generate` 的排队、运动描述、参考图对齐与失败重试；
- 静图生成通常 **更快、更稳**；
- 脚本已给 **0–5s、5–10s…** 时间窗，VO 总长可塞进总时长，**色条/定长 overlay** 即可填满时间轴，模型认为已满足「时间结构」层面的交付。

### 4. 技能 / 系统边界未强制「必须运动视频」

若 `generation-skill` / `creative-skill` / `assembly-skill` 未明确禁止「仅用静图 overlay 替代分段生成视频」的交片方式，模型会在多种可行路径中选 **更轻** 的一条。

---

## 解决办法

### A. 技能与系统提示（推荐，治本）

1. **明确触发条件**  
   - 当用户提到「短视频 / TikTok / Reels / 多场景视频」且**未**写明「只要静态信息图 / PPT 风」时：**默认**每段视觉应为 `video_generate`（或图生视频），禁止仅用 `image_generate` + 全片 overlay 作为主路径交片。

2. **与用户稿措辞对齐**  
   - 若 brief 使用「Picture to show」，规则中写明：须向用户确认是 **(1) 纯静态配图** 还是 **(2) 需要运镜/动效**；未确认前，对「短视频」类需求 **偏向 (2)**。

3. **assembly 前置条件**（若 assembly-skill 已有 prerequisites，可收紧）  
   - 要求「分段视频素材」时：若上游只有静图，须先经 `video_generate`（如 reference2video / text2video）或用户明确豁免。

4. **产品侧默认策略**（若可配置）  
   - 项目类型或模板选择「口播科普短视频」时，默认打开「强制分段视频」或提高 `video_generate` 优先级。

### B. 用户侧（治标、见效快）

在需求里直接写清，例如：

- 「每段需要 **AI 生成运动视频**（运镜 / 动效），不要用纯静态图拼时间线。」
- 「视觉可以是图生视频：先用图定风格，再 `video_generate`。」

### C. 观测与回归

- Langfuse 中对该类 case 打点：是否出现 `video_generate`、是否出现「slide-style」组装的 assistant 文本，便于统计策略漂移。
