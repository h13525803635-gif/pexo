## 三、问题清单（按严重度）

### 🔴 P0：跳过路由层 + 模型/参数全错（核心根因）

#### 3.1 generation-skill / script-skill 一次都没打开

系统提示词明确归属：

> **generation-skill**: ... Handles **model routing (Seedance / Kling / Grok — four-layer system)**, prompt writing, single/multi-sequence generation, and post-generation sanity checks.
> **script-skill**: ... production model routing (four-layer system) ...

但本 trace 中 `read_file` 只加载：

```
assembly-skill/SKILL.md           ×2（且是 edit_video 报错后被逼着才读）
brainstorm-skill/SKILL.md         ×1
brainstorm-skill/references/image-generation-guide.md   ×1
```

**generation-skill 与 script-skill 全程未读**。模型选择是凭先验拍脑袋。

#### 3.2 三次 video_generate 全部参数错配

```python
video_generate(
  provider='kling',               # ❌ 应为 seedance
  model='kling-v3-omni',
  mode='reference2video',         # ❌ 非 lipsync 模式
  provider_param={
    'kling_reference2video': {
      'prompt': '... talking to camera like a live streamer ...',
      'aspect_ratio': '9:16',
      'duration': '6',
      'image_list': [ai_host_ref, product_imgs],
      'sound': 'on',              # ❌ 既然有独立 TTS，按 96933223677 应为 "off"
      # ❌ audio_list 完全缺失 → 嘴型零驱动
    }
  }
)
```

| 项 | 实际 | 应该 |
|---|---|---|
| provider | `kling` | `seedance` |
| audio_list | 缺失 | 传入 TTS 切片 |
| sound | `"on"` | `"off"`（或承担 lipsync 唯一音源） |

注：`kling-v3-omni` 名字带 omni 暗示模型本身可能支持 audio-driven lipsync，但 Agent 用的是 `kling_reference2video` 这个 provider_param key（非 lipsync key），且未传 audio。

### 🔴 P1：检测到英文 co-gen 后仅做最小修补

obs 26（`analyze_file_content`）返回：

> "The video does not contain any spoken Chinese dialogue. It features a young woman speaking in English, saying ..."

Agent 在 obs 77 的回复："视频里是英文台词（由模型co-gen），我们要叠加中文配音VO，需要把视频原声音量压低"——把视频音量压到 **0.2**（仍可听），而不是：

- ❌ 重新生成视频（应改 `sound:"off"` 或喂中文 TTS 进 audio_list）
- ❌ 视频音量直接归零
- ❌ 告知用户"嘴在说英文 vs 配音是中文"的根本矛盾

**这是 case 96933223677 双音轨问题的精确复现**：成片自带英文 co-gen 人声（vol 0.2）+ 独立中文 TTS（vol 1.0）同播。

### 🔴 P1：字幕手算时间，违反 assembly-skill 硬规则

`audio_produce` 返回了 `chinese_voiceover_subtitle_20260513T013736_ac3a3918.json`（ElevenLabs 字级对齐 JSON），是字幕时间轴的 ground truth。

但 Agent 在最终 `execute_edit_video` 里用的是：

```python
  {start_ms: 4200,  end_ms: 8000,  text: '这两款就够了！'},

  {start_ms: 19500, end_ms: 24000, text: '护肤选对了，才事半功倍！'}

```

assembly-skill 第 125 / 127 条明确规定：

> "When `audio_produce` returns a `subtitle_file`... that file's timestamps are the ground truth for subtitle timing and **must** drive the final subtitle layer."
> "If a `subtitle_file` exists, manually constructing `kind=text` clips with hand-calculated `start_ms`/`out_ms` values is **blocked**."

Agent 走的正是规则里 blocked 的兜底路径。

## 四、关键失败链

```
用户口播需求
  ↓
Agent 跳过 generation-skill / script-skill 路由层
  ↓
凭先验选 kling-v3-omni reference2video（默认产线，不是口播产线）
  ↓
不传 audio_list（嘴型零驱动）+ sound:"on"（让模型自己 co-gen 英文人声）
  ↓
TTS 独立做（产线分裂：嘴 vs 中文配音 vs 残留英文 = 三套互不相干的时间轴）
  ↓
装配阶段没归零视频音轨（保留 0.2 英文人声）
  ↓
字幕手算时间（无视 ElevenLabs alignment json，违反 assembly-skill 硬规则）
  ↓
时长不够用同一段重复 5s 凑齐
  ↓
交付时一句风险都不提
```

## 六、修复建议

### 6.1 Skill 层

1. **brainstorm-skill 入口**：识别"主播 / 口播 / 数字人 / talking head" 关键词时，强制触发"必须读 generation-skill"的前置 checklist；并明确禁止把"口播视频 + TTS"路由到 kling reference2video。
2. **generation-skill / video-models-scheduling**：把 04453569710 的产线规则升级为 **hard rule**：用户场景为口播且已有 TTS → 必须 Seedance + audio_list + sound:"on"，禁止任何其他降级方案。
3. **assembly-skill**：sound-on 视频轨在存在独立 VO 时**默认 vol=0**，仅当成片人声是唯一口播且无 TTS 时才允许保留。
4. **audio-aware mixing 前置依赖**：当 `analyze_audio` 失败时禁止进入 mixing，必须先解决文件路径问题。

### 6.2 用户告知层

5. **brainstorm-skill / 交付模板**：当实际执行偏离了用户口语理解（"对口型主播视频" → 实际"主播形象 + 旁白"）时，最终回复必须强制 disclose 差异，不能仅描述功能性结果。
