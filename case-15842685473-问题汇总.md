## 2. 核心矛盾（用户体感）

1. **改动量小**：用户每次仅修正**部分拼写**（或局部文案），主观上属于「小改」。  
2. **成本高**：系统侧多次触发**整段/整片级**生成（视频 + 音轨等），积分按次累加，用户认为**不值**。  
3. **语言质量预期**：Pexo 对斯洛文尼亚语**变音符号、重音**等处理不够精准，用户为「听得对」反复改字，进一步放大「小改大扣」的感受。

---

## 3. 问题拆解（现象 → 归因方向）

| 编号 | 现象 | 可能归因 |
|------|------|----------|
| A | 两次修改仍各出一条完整 96s 成片 | 产品/编排：**全片重算**为主路径，缺少「仅修补音轨/单段」 |
| B | 积分随版本线性放大 | 计费：**缺少**「文案小 diff / 同项目二改」梯度或修补价 |
| C | 重音与特殊字母不准 | 模型/管线：**小语种 TTS** 能力边界；缺少文本规范化与读法辅助 |
| D | 用户不理解「改两个字」为何接近全价 | 体验：**未充分说明**本次操作会触发哪些计费步骤 |
| E | 若流程含「带字图 → 视频」 | 创意 skill：**图上字与口播稿**若未拆分，易迫使整 ref + 整段重出 |

## 4. 已讨论过的改进方向（速查）

### 4.1 产品 / 计费

- 区分**全片重生成** vs **仅 TTS / 单段 / 剪辑**；对「仅脚本文案小变更」设**修补价**或阶梯计价。  
- 修改前展示**积分预估**与**将执行步骤**（全片 or 仅音轨等）。  
- 项目结束提供**简单账单摘要**（几次全片、几次 TTS、几次剪辑），便于客服与用户对齐预期。

### 4.2 技术与管线

- **句级/段级**合成与缓存：只重跑变更句对应音频（及必要画面），其余段复用。  
- **文本规范化**再进 TTS：Unicode 规范化、数字/缩写展开、词典纠错，减少「同一意思反复改拼写」。  
- 评估/加强 **sl** 语系 TTS 或提供**发音词典 / 自定义读法**（品牌名、地名等）。  
- **字幕与口播解耦**：关键信息以字幕为准，降低「为图上错字」整段重做的概率。

### 4.3 体验与流程

- **口播稿锁定**步骤（尤其小语种）：高亮可疑片段，用户确认后再合成。  
- **单句/单段 TTS 试听**，再合并成片，避免一上来就扣满额长视频成本。  
- 小改时提示：**听感可能几乎不变，是否仍要全片重生成**（若业务允许「仅字幕更新」则给选项）。

### 4.4 Agent / Skill（创意与修改）

- **creative-skill**：明确「图上字」与「口播/定稿脚本」分工；handoff 写清 ref 管构图/风格而非强制保留所有 baked-in 文字。  
- **modification-skill**：优先**局部重拍/剪辑**，避免默认全量 v2。  
- **case-analyzer skill**：分析报告中固定增加**迭代间脚本文本 diff**、**从工具调用推断的计费相关事件**，便于复盘此类 case。

### 4.5 客服与策略

- 对「仅拼写微调却多次全价」类争议，可设**规则化补偿**（券/积分），并写入帮助中心，稳定预期。
