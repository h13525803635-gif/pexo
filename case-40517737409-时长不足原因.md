# Case 40517737409 — 目标 2 分钟 / 约 40 秒，成片约 82 秒 / 25 秒 — 原因说明

依据 Langfuse 拉取的 trace（`trace-2` / `trace-3` 等）与 `video_generate`、`video-editor__execute_edit_video` 工具回包整理。

---

## 结论（一句话）

**主要是「根本没按目标总秒数排够 `video_generate` 次数与每段 `duration`」；其次有「整轨音频不能进 Seedance r2v」与「全片合成时误用 120 秒裁切」叠加。**

---

## 1. 前半「约 2 分钟」为何只有 ~82 秒

- 成片 `telemetry_first_half_v1.mp4`：**6 条视频**，工具返回 **`duration`: 82.0s**。
- 对应并行 `video_generate` 的 **`duration` 参数之和**：**15 + 14 + 15 + 15 + 15 + 9 ≈ 83 秒**（与成片一致，差异来自剪辑/crossfade）。
- Seedance 单段上限约 **≤15s**，要接近 **120s** 至少需要约 **`ceil(120/15)=8` 次**生成；实际只做了 **6 次**，因此**从计划上就没有生够**。
- Agent 在创意层用「乐曲 0:00～2:04」做叙事映射，但**未在工具层做硬性校验**：`Σ(duration) ≥ 用户目标成片时长`。

---

## 2. 后半「约 40 秒」为何体感约 25 秒

- 后半按 **Seq G / H / I** 共 **3 段**规划（每段仍约十几秒档），名义上奔着约 40s+。
- 全片 `telemetry_full_v1.mp4`：**9 段**，工具返回 **`duration`: 107.0s**（非 120+40）。
- Trace 中 Agent 曾自述：9 段 raw 总长会超过其**误判的「120 秒制約」**，因而**对各段 `in/out` 大幅 trim**；乐曲实际约 **151.87s**，该「120s」并非来自用户或 API 的可靠硬约束。
- **107 − 82 ≈ 25s**：相对已交付前半，后半在时间线上净增量约 **25 秒**，与用户「再要约 40 秒」的落差由此产生。

---

## 3. API 侧：整轨 BGM 进 `audio_list` 会失败

- 首批 `video_generate` 带整首 MP3 时，接口报错大意：**`audio_list` 音频时长须 ≤ 15.2 秒**（Seedance r2v）。
- 后续改为「视频不绑整轨、BGM 后期 overlay」合理，但若不在计划里**显式按秒拆段并对齐**，更容易出现「叙事对齐乐曲、成片秒数对不齐」。

---

## 4. 根因分层（便于改 skill / 改 prompt）

| 层级 | 内容 |
|------|------|
| **执行** | 缺「总时长预算校验」：`Σ video_generate duration` 与 `execute_edit_video` 输出时长应对齐用户目标；避免虚构「120s」裁切。 |
| **流程** | 长 MV = 多段拼接；每段 ≤15s 时，**段数 = 目标秒数 / 15 向上取整**（再加 crossfade 余量）。 |
| **模型/API** | 单段上限 + `audio_list` ≤15.2s → 不能靠单次长音频锁整首；需多段 + 后期或分段音频切片。 |

---

## 5. 与「早期版本误差更小」的关系

本 case 误差主要来自 **段数规划与错误裁切**，不是「单段偶尔少几帧」能解释的量级；若早期版本有更严的时长对账或更少误用约束，体感会更好。

---

## 6. 为什么会出现这种情况（归纳到机制）

**本质：用户锁定的「总秒数」没有在系统里被落实为「足够多的生成任务 + 正确的合成时间线」，且缺少自动校验与交付门禁。**

叠加上本 case 的具体机制包括：

1. **分段规划不足**：长视频必须多段生成；未按 `ceil(目标总时长 / 单段上限)` 反推段数时，**从第一下规划就会短一截**。
2. **叙事时长 ≠ 工具时长**：创意里可以对齐乐曲结构，但若没有强制步骤把「每段 `duration`」加总并对齐用户确认的 `T`，就会出现「故事很长、成片很短」。
3. **接口约束未前置进编排**：例如 r2v 对 `audio_list` 有短音频上限；整轨失败后改「后期铺 BGM」若不做按秒对表，**音乐与总时长都更容易飘**。
4. **合成阶段误用规则**：在不存在可靠硬约束时，对 timeline 做大幅 trim，会在已偏短的基础上**再砍一刀**。
5. **无导出前校验**：未检查 `输出时长 ≈ 用户目标（±容差）`，问题会直接到用户端。

---

## 7. 我们应该如何避免（可落地产研动作）

### 7.1 流程：长单强制「时长预算表」

- 用户确认总时长 `T` 后，**必须先产出**：段序号、每段秒数、选用模型/单段上限、转场/crossfade 扣减、是否与 BGM 锁拍。
- **硬规则**：在 `Σ 各段有效时长 − 转场损耗 ≥ T` 成立之前，**不启动批量 `video_generate`**。

### 7.2 Agent / Skill：自检与禁止项

- **自检**：排完每批 `video_generate` 后计算 `sum(duration)`；若 `< 目标 − 容差`，必须**自动加段/拆段**，禁止「少生凑合交」。
- **禁止**：无用户明文时，**不得**用任意「默认最大成片时长」裁整条时间线。
- **音频**：长 BGM 默认**不进 r2v `audio_list`**；若要对节拍/情绪，使用 **≤ 接口上限的切片** 或纯后期对齐，并写入预算表。

### 7.3 工程：硬门禁（比纯提示词可靠）

- **`execute_edit_video` 前**：校验 timeline 总时长；低于目标则 **fail closed**（返回错误给 Agent 重排），禁止静默导出明显短于目标的成片。
- **交付用户前**：用 ffprobe/元数据校验；可选与用户在对话里锁定的 `T` 对比，不通过则走「补段」或人工审核队列。

### 7.4 产品：关键节点显式确认

- 长视频开做前展示：**预计段数、预计总时长、与目标的差距**；用户点确认后再跑高成本生成。
- 若必须裁切（真有硬上限），**必须说明裁哪段、裁多少秒、对叙事影响**，由用户确认，而非模型自裁。

### 7.5 观测与迭代

- 在 Langfuse/日志打点：`target_duration`、`planned_sum_duration`、`rendered_duration`、`trim_reason`。
- 对「目标 > 60s」项目做指标看板：**短于目标的比例、平均缺口**，驱动 skill 与门禁持续迭代。

## 8. 对用户的沟通建议（内部归因 vs 对外话术）

**内部技术归因**可写清：Seedance 单段上限、本 case 仅 6 段而非至少约 8 段、合成误裁、`audio_list` 上限等（见上文 §1–§4、§6）。

**对终端用户不宜逐条复述上述实现细节**（易引发「你们在甩锅模型」或不必要的模型名/段数争论）。建议对外统一落在：

- **结果**：成片时长**未达**您确认的目标，属于我方交付问题。  
- **原因（通俗）**：长视频需**分段制作与组装**；本次在**编排与时长校验**上出现疏漏，导致实际生成与剪辑后的总时长偏短。  
- **处置**：按您确认的总时长**重新编排分段并补做**（或按客诉政策补偿），直至对齐。

若需更笼统的一句：**「组装与时长编排未达标，将重新编排后交付。」**  
避免单独甩锅「用户只传了音乐」——音乐形态会加大编排难度，但**达标责任仍在产品与流水线**。

---

## 9. 风险提示（与输入形态相关，供产研留意）

团队内讨论提到：当用户侧 **以整曲 / 音乐驱动为主**、画面叙事需强锁音乐节拍时，若未先做「时长预算表 + 切片策略」，分段与对表更容易失败（本 case 亦涉及整轨 BGM 与 r2v `audio_list` 限制后的改策略）。

**规避**：长 BGM 场景仍适用 §7.1 预算表、§7.2 音频策略、§7.3 导出门禁；不把「只有音乐」当作对用户的说辞。
